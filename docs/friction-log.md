| Deploy app to CMX k3s cluster from Apple Silicon Mac | `docker build` + `docker push`, then `helm install` — app runs on the cluster | Pod crashlooped with `exec format error`. Docker built an ARM image by default (Apple Silicon) but the CMX cluster runs AMD64. No warning from Docker or Kubernetes about the architecture mismatch — just a cryptic exec format error. Hit this TWICE — once with `docker build --platform`, again when using `go build` inside Dockerfile without `GOARCH=amd64`. | ~10 min each time | First time: rebuilt with `docker build --platform linux/amd64`. Second time: added `GOARCH=amd64` to the `go build` line in Dockerfile. Permanent fix: bake `GOARCH=amd64` into the Dockerfile so `make push` always produces an AMD64 image. | Annoyance — common gotcha but the error message could be clearer. Recurring because there are multiple places to fix (Docker platform vs Go cross-compile). |
| Access app externally via CMX exposed port | Expose port 8080, get a hostname, app is accessible | Got 503 Service Unavailable. CMX requires a NodePort service but the default Helm service type is ClusterIP. No error message indicated this — just a generic 503. | ~15 min | Found in CMX networking docs that port exposure uses NodePort, not ClusterIP. Changed service type to NodePort via `helm upgrade --set service.type=NodePort`. | Blocker — CMX UI accepted the port exposure without warning that the service type was incompatible |
| Set up custom domain with TLS on CMX cluster | Point CNAME to CMX hostname, enable ingress + cert-manager, get HTTPS on custom domain | Multi-step runaround: (1) Had to expose the app's NodePort first, then realize we also need Traefik's NodePort exposed separately for ingress/TLS to work. (2) CNAME records had to be updated multiple times as each new CMX port exposure generates a different hostname. (3) Need to understand the relationship between CMX port exposure, k3s Traefik ingress controller, and cert-manager HTTP-01 challenges — none of this is documented together. | ~30 min (ongoing) | Exposed Traefik's HTTP NodePort (31424) via CMX, created Let's Encrypt ClusterIssuer, enabling Helm ingress with tls.mode=auto. Required updating CNAME records three times. | Blocker — too many moving parts with no unified guide for "custom domain + TLS on CMX" |
| cert-manager HTTP-01 challenge failing with 409 on CMX k3s cluster | cert-manager creates solver ingress, Let's Encrypt validates challenge, cert is issued | Two issues: (1) cert-manager's ingress-shim controller was disabled, so the ingress annotation didn't auto-create a Certificate — had to create it manually. (2) The ACME HTTP-01 solver ingress was created without an `ingressClassName`, so Traefik ignored it and Let's Encrypt got a 409 instead of the challenge response. | ~10 min (ongoing) | Manually patched solver ingress with `ingressClassName: traefik`. Root cause: cert-manager doesn't know which ingress class the solver should use when the cluster has a non-default setup. | Blocker — silent failure, no error in cert-manager logs pointing to the missing ingressClassName |
| Let's Encrypt cert for custom domain on CMX cluster | HTTP-01 challenge completes, valid cert issued for app.astrid.fit | Impossible through CMX networking. CMX routes traffic through Cloudflare's proxy, which only terminates TLS for `*.ingress.replicatedcluster.com`. Cloudflare returns error 1001 when the Host header is a custom domain. DNS-01 challenge is the alternative but requires DNS provider API access (Namecheap doesn't offer this on basic plans). | ~45 min total across all TLS attempts | Pivoted to using CMX's auto-provisioned cert on the `.replicatedcluster.com` hostname for valid TLS, and self-signed cert for the custom domain. Cannot get a real Let's Encrypt cert for a custom domain on CMX without DNS-01 or a non-proxied cluster. | "I would have churned" — spent significant time trying to make TLS work for a custom domain on CMX with no documentation indicating this is unsupported |
| `helm upgrade` with new image tag | App redeploys with new image, existing settings (service type, ingress, etc.) preserved | `helm upgrade` without re-specifying `--set service.type=NodePort` reverts the service back to `ClusterIP` (the chart default). App becomes unreachable on CMX with a generic "Service Unavailable" error. Hit this twice — easy to forget when focused on deploying a code change. | ~5 min each time | Re-ran `helm upgrade` with `--set service.type=NodePort`. Long-term fix: use a `values-cmx.yaml` overlay with `-f` flag so CMX-specific settings are always applied. | Annoyance — Helm's behavior is correct (values.yaml is the source of truth) but it's a footgun when iterating quickly |
| Switching from `helm upgrade --set` to `helm upgrade -f values-cmx.yaml` | Clean upgrade using the overlay file | Helm rejected the upgrade with "Apply failed with N conflicts" on the Ingress and Certificate resources. Mixing `--set` overrides (from earlier upgrades) with `-f` overlay (on later upgrades) causes Helm field ownership conflicts. The error message lists the conflicting fields but doesn't suggest a fix. | ~5 min | `helm upgrade --force-conflicts` to force Helm to take ownership of the conflicting fields. | Annoyance — Helm's server-side apply ownership tracking is doing its job, but the error is confusing when you're just trying to standardize on an overlay file |
| Redeploy after code changes — cascading misconfiguration | Push new image, helm upgrade, app is accessible at same URL | What should have been a simple image update turned into ~45 min of debugging across multiple wrong turns. The core issue: Traefik has two entrypoints (web/HTTP on NodePort 31424, websecure/HTTPS on NodePort 32424). An ingress with a TLS section routes ONLY on websecure, but the CMX port exposure was mapped to web (31424). Instead of identifying this mismatch, multiple speculative fixes were applied — disabling TLS, changing hostnames, restarting Traefik, trying `--force` flags — each making things worse. Key lesson: Traefik entrypoint + CMX port mapping + ingress TLS config are a triple that must be consistent. TLS ingress = websecure entrypoint = port 32424. No TLS = web entrypoint = port 31424. | ~45 min | Exposed Traefik's websecure NodePort (32424) via `replicated cluster port expose 2db581de --port 32424 --protocol https` → new hostname `tender-ride.ingress.replicatedcluster.com`. Updated `values-cmx.yaml` and Cloudflare CNAME for `app.astrid.fit`. Final stable CMX config: ingress with TLS (selfsigned) → Traefik websecure (32424) → CMX port exposure on 32424. Previous hostnames `vibrant-dijkstra` (31424/HTTP) and `quizzical-ellis` (31110/wrong port) are stale. | Blocker — CMX + Traefik + Helm TLS interaction is underdocumented. No single error message pointed to the entrypoint mismatch. Resulted in 4 separate CMX port exposures and 3 Cloudflare CNAME updates. |
| Custom domain (app.astrid.fit) via CNAME to CMX cluster | CNAME app.astrid.fit → tender-ride.ingress.replicatedcluster.com, site loads over HTTPS | Three separate failures across ~16 hours: (1) With Cloudflare proxy enabled (orange cloud): error 1014 "CNAME Cross-User Banned" — Cloudflare blocks cross-account CNAME chains through its proxy. (2) With Cloudflare proxy disabled (grey cloud / DNS-only): 403 from Replicated's Cloudflare — the request reaches Replicated's proxy with `Host: app.astrid.fit` but their Cloudflare only recognizes `tender-ride.ingress.replicatedcluster.com`, so it rejects it. (3) Replicated has no "custom domain" feature for CMX cluster ingress hostnames — their custom domains feature only covers registry, proxy, download portal, and enterprise portal. The CNAME approach is fundamentally broken for CMX. | ~2 hours across two days | Deployed a Cloudflare Tunnel (`cloudflared` pod in the cluster). The tunnel creates an outbound connection from the cluster to Cloudflare's edge, completely bypassing Replicated's Cloudflare proxy. Traffic flow: browser → Cloudflare (our account) → tunnel → cloudflared pod → astrid:8080. Added public hostnames for app.astrid.fit and www.astrid.fit in the Zero Trust dashboard under Published application routes. Cloudflare auto-creates the correct proxied CNAME records. | "I would have churned" — no documentation anywhere (Replicated or Cloudflare) warns that CNAMEing to a `*.replicatedcluster.com` hostname won't work with a custom domain. The Cloudflare error codes (1014, 403) don't suggest the root cause. The only viable solution (Cloudflare Tunnel) requires knowledge of a completely separate product (Cloudflare Zero Trust). |
| `create-release` GitHub Action — passing chart directory instead of `.tgz` | Pass `chart: chart/astrid` (directory path), action packages and uploads it | Action errored: "Chart chart/astrid is a directory, not a file". The `create-release` action only accepts a packaged `.tgz` file, not a chart directory. This is not obvious from the action's README — the `chart` input description doesn't specify the format. The Replicated CLI's `.replicated` config flow auto-packages directories, but the GitHub Action does not. | ~10 min | Added a `helm package chart/astrid -d .` step before `create-release`, passing `astrid-0.1.0.tgz`. | Annoyance — the `chart` input name suggests it accepts a chart (directory or file), and the error only surfaces at runtime. The action README should state "path to a packaged .tgz file". |
| `create-release` GitHub Action — `apps is not iterable` error | Action looks up app by slug and creates a release | Cryptic JavaScript error: "apps is not iterable". The service account API token's RBAC policy was missing `kots/app/*/read` permission. The action tried to list apps to resolve the slug to an ID, got a 403, and the error handler produced a confusing JS error instead of "permission denied". | ~10 min | Added `kots/app/*/read` to the `ci-automation` RBAC policy. | Annoyance — the error message is a leaked JS implementation detail. Should say "Insufficient permissions to list apps" or similar. |
| Replicated release promotion — version label must exactly match Chart.yaml version | Promote release to Unstable with `version: 0.1.0+<sha>` (valid semver) while chart has `version: 0.1.0` | Same error twice, for two different reasons. First attempt: raw git SHA as version label — fails because channels have `semverRequired: true` by default. Second attempt: `0.1.0+<sha>` (valid semver) — still fails because the Replicated API requires the version label to **exactly match** the `version` field in `Chart.yaml`. The error message is the same both times: "Version label does not match any Helm charts in the release." This is doubly misleading: (1) it doesn't mention semver on the first failure, (2) it doesn't explain the exact-match requirement on the second failure. Had to read the `replicated-lib` source code to discover the exact-match constraint — the docs don't spell this out clearly. | ~30 min across two attempts | Stamp `Chart.yaml` version in CI before packaging: `sed -i "s/^version:.*/version: 0.1.0+<sha>/"`, then pass the same version to `create-release`. The chart `.tgz` filename also changes to match. | Blocker — the same vague error message for two different root causes made this a two-round debugging cycle. Docs should state: "The version label must exactly match the `version` field in one of the Chart.yaml files in the release." |
| `helm dependency build` fails in CI — Bitnami repo not registered | CI runs `helm dependency build chart/astrid`, subchart dependencies resolve | Error: "no repository definition for https://charts.bitnami.com/bitnami". Works locally because `helm repo add bitnami ...` was run previously. CI runners have a clean Helm environment with no repos configured. The deprecated `https://charts.bitnami.com/bitnami` URL requires `helm repo add` but OCI references (`oci://registry-1.docker.io/bitnamicharts`) do not. | ~10 min | Switched `Chart.yaml` dependencies from `repository: "https://charts.bitnami.com/bitnami"` to `repository: "oci://registry-1.docker.io/bitnamicharts"`. OCI pulls work without `helm repo add`. | Annoyance — easy to miss because it works locally. Bitnami's migration from Helm repo to OCI means charts using the old URL break on clean CI runners. |
| Application CR (`kots.io/v1beta1`) fails on plain k3s cluster | `helm install` with the Replicated SDK subchart deploys successfully | Error: "no matches for kind 'Application' in version 'kots.io/v1beta1' — ensure CRDs are installed first." The `Application` custom resource is a KOTS-specific CRD that only exists on clusters with KOTS installed (or embedded cluster). Plain k3s/CMX clusters don't have it. | ~10 min | Wrapped the manifest in `{{- if .Capabilities.APIVersions.Has "kots.io/v1beta1" }}` so it only renders on KOTS clusters. | Annoyance — the Replicated docs don't warn that the Application CR template needs a guard for non-KOTS installs. Easy to hit when testing Helm-only installs. |
| Replicated SDK crashloops without a license in CI | Add SDK subchart, CI deploys to CMX test cluster, all pods ready | `helm install --wait` times out after 5 min. The SDK pod needs `global.replicated` values (injected by the Replicated registry) to authenticate. Direct `helm install` from a local chart has no license data, so the SDK fails to start. No error message — just a silent timeout. | ~10 min | Added `--set replicated.enabled=false` to CI deploy steps to skip the SDK in test clusters. | Annoyance — the SDK silently fails without license data. Would be helpful if the SDK logged a clear "no license found, cannot start" message instead of crashlooping. |
| Image proxy (`images.astrid.fit`) requires license auth — breaks CI | Proxy all images through custom domain for rubric 2.2, CI deploys to test cluster | `helm install --wait` times out. All images (app, postgresql, redis, busybox init container) are now proxied through `images.astrid.fit/proxy/astrid/...` which requires Replicated license auth to pull. CI test clusters have no license, so every image pull fails silently. Took two rounds of timeout debugging to identify — first round fixed the SDK, second round identified the image proxy as the remaining cause. | ~20 min across two CI runs | Override all image references back to direct registries in CI deploy steps: `--set postgresql.image.registry=docker.io`, `--set redis.image.registry=docker.io`, `--set initImage=busybox:1.36`. Made the init container image a configurable value. | Blocker — `helm install --wait` gives zero diagnostic output when pods fail to pull images. Had to reason about what changed (proxy images) and what CI lacks (license). Adding `kubectl get events` or `kubectl describe pods` after a timeout would help enormously. |
| Custom proxy domain CNAME verification fails on Cloudflare | Add `images.astrid.fit` as custom proxy domain in Vendor Portal, create CNAME to `proxy.replicated.com` | Error in Vendor Portal: "custom hostname does not CNAME to this zone." Replicated uses Cloudflare for SaaS under the hood. When your domain is also on Cloudflare with the proxy enabled (orange cloud), Cloudflare resolves the CNAME to its own IPs instead of `proxy.replicated.com`. Replicated's Cloudflare zone can't see the CNAME target. | ~15 min | Switch the CNAME record to **DNS only (grey cloud)** in Cloudflare. Also verify CNAME Flattening is set to "Flatten at root only" (the default). | Annoyance — neither Replicated nor Cloudflare docs mention this interaction. The error message doesn't suggest checking proxy status. Common gotcha for anyone using Cloudflare DNS. |
| Replicated pull secret not auto-wired into chart pods | Install from Replicated registry, SDK creates `enterprise-pull-secret`, all proxied images pull successfully | Pods fail with `ImagePullBackOff`. The SDK creates the pull secret, but the chart's own deployments and Bitnami subcharts don't reference it. The `imagePullSecrets: []` default in values.yaml means no pods get the credential. The Replicated docs describe a helper template pattern for `_helpers.tpl` that checks for `global.replicated.dockerconfigjson` and auto-includes `enterprise-pull-secret`, but this isn't mentioned in the SDK installation guide or the image proxying docs. For Bitnami subcharts, you also need `global.imagePullSecrets` since they read pull secrets from the global scope. | ~20 min | Added `astrid.imagePullSecrets` helper to `_helpers.tpl` that auto-detects the Replicated pull secret. Set `global.imagePullSecrets` for Bitnami subchart compatibility. | Blocker — the SDK creates the secret but nothing tells you to wire it into your chart. The install succeeds (Helm doesn't check image pulls), then pods fail. Should be a prominent step in the SDK installation guide: "Add the imagePullSecrets helper to your chart." |
| Image proxy fails with 400/404 — external registry not configured | Proxy all container images through `images.astrid.fit/proxy/astrid/...`, pods pull successfully | Every image pull fails: busybox gets 404 on the token endpoint, bitnami images get 400 Bad Request. Errors occur even with `proxy.replicated.com` directly (not just the custom domain), and even with the `enterprise-pull-secret` attached. The Replicated SDK pod pulls fine from `proxy.replicated.com/library/...` because that's a different path for Replicated's own images. | ~30 min debugging across multiple attempts | **Must add Docker Hub as an external registry in the Vendor Portal** (Images > Add External Registry > DockerHub, hostname `index.docker.io`, with credentials). The proxy cannot proxy images from a registry it has no credentials for, even for public images. Also: image paths should omit `docker.io/` prefix and `library/` namespace — use `proxy/astrid/busybox:1.36` not `proxy/astrid/docker.io/library/busybox:1.36`. | Blocker — this prerequisite is not surfaced in the custom domains setup flow or the image proxying quick-start. The 400/404 error messages don't hint at the cause. Easy to assume the proxy "just works" for public images once the domain is verified. The external registry setup should be a mandatory step in the custom domains wizard, or the error should say "no registry credentials configured for docker.io". |
